## 1. 프로젝트 개요

### 1.1 EvalVault란 무엇인가?

**EvalVault**는 RAG(Retrieval-Augmented Generation) 시스템의 **품질 측정 · 관측 · 개선**을 한 번에 처리하는 통합 평가 플랫폼입니다.

### 1.2 핵심 가치 제안 (Value Proposition)

```
┌─────────────────────────────────────────────────────────────────────────────────────┐
│                         EvalVault 핵심 가치                                       │
├─────────────────────────────────────────────────────────────────────────────────────┤
│  단순 점수 계산기가 아닌, "RAG 시스템의 품질을 데이터셋/메트릭/트레이싱   │
│  관점에서 일관되게 관리하는 운영 콘솔"                                          │
│                                                                              │
│  평가 → 자동 분석 → 보고서/아티팩트 저장 → 비교가 하나의 run_id로 끊김 없이     │
│  이어져서, 재현성과 개선 루프가 매우 빠름                                     │
└─────────────────────────────────────────────────────────────────────────────────────┘
```

### 1.3 다섯 가지 핵심 축

| 축 | 설명 | 구현 |
|----|------|------|
| **평가(Evaluation)** | 데이터셋 기반 다양한 LLM/리트리버/프롬프트 조합 실험 | Ragas 메트릭 + 커스텀 메트릭 |
| **관측(Observability)** | Stage 단위 이벤트/메트릭, Langfuse/Phoenix 트레이싱 | `StageEvent`, `StageMetric`, TrackerPort |
| **표준 연동(Open RAG Trace)** | OpenTelemetry + OpenInference 스키마 | OpenInference 이벤트, span 속성 |
| **학습(Domain Memory)** | 과거 실행에서 도메인 지식/패턴 축적 | Factual/Experiential/Behavior 레이어 |
| **분석(Analysis Pipelines)** | 통계·NLP·인과 모듈 포함 DAG 파이프라인 | `PipelineOrchestrator`, `AnalysisModulePort` |

### 1.4 주요 사용자 시나리오

| 사용자 | 주요 작업 | 제공 인터페이스 |
|--------|----------|----------------|
| **개발자** | 평가 실행, history/compare/export | CLI (`evalvault run`) |
| **평가 담당자** | 평가 실행/업로드, 히스토리 탐색, 보고서 생성 | Web UI (Evaluation Studio) |
| **운영팀** | Phoenix 기반 드리프트 감시, Gate 실행 | CLI + Phoenix 연동 |

## 2. 아키텍철적 분석

### 2.1 적용 설계 방법론

EvalVault는 세 가지 핵심 아키텍처 방법론을 통합하여 설계되었습니다:

| 방법론 | 원작자 | 핵심 원칙 | EvalVault 적용 |
|--------|--------|-----------|----------------|
| **Hexagonal Architecture** | Alistair Cockburn | 도메인 로직을 외부 의존성으로부터 격리 | 포트/어댑터 패턴 |
| **Clean Architecture** | Robert C. Martin | 의존성 규칙, 계층 분리 | 도메인 → 포트 → 어댑터 |
| **Domain-Driven Design** | Eric Evans | 도메인 중심 설계, Rich Domain Model | 엔티티, 서비스, 메트릭 |

### 2.2 의존성 방향

```
의존성 방향: 외부 → 내부
┌─────────────────────────────────────────┐
│  Adapters (외부 계층)                   │
│  └─> Ports (인터페이스)                 │
│      └─> Domain (핵심 로직)             │
└─────────────────────────────────────────┘
```

**규칙**:
- 어댑터는 포트에 의존
- 포트는 도메인에 속하지만 도메인 서비스에 의존하지 않음
- 도메인은 포트에만 의존 (어댑터에 직접 의존하지 않음)

### 2.3 계층별 상세 구조

#### Domain Layer (도메인 계층)
- **책임**: 핵심 비즈니스 로직
- **특징**: 완전히 독립적, 외부 프레임워크/라이브러리에 의존하지 않음
- **주요 구성요소**:
  - `entities/`: Dataset, EvaluationRun, AnalysisBundle, Experiment, BenchmarkRun, ImprovementReport, KnowledgeGraph, RAGTraceData, Memory
  - `metrics/`: 도메인 특화 메트릭 (InsuranceTermAccuracy, EntityPreservation)
  - `services/`: RagasEvaluator, ExperimentManager, AnalysisService, PipelineOrchestrator, ImprovementGuideService, MemoryAwareEvaluator

#### Ports Layer (포트 계층)
- **책임**: 도메인과 외부 세계 사이의 계약(Contract) 정의
- **특징**: Protocol/ABC 기반 인터페이스
- **주요 포트**:
  - Inbound: EvaluatorPort, AnalysisPipelinePort, WebUIPort
  - Outbound: LLMPort, DatasetPort, StoragePort, TrackerPort, DomainMemoryPort, AnalysisModulePort

#### Adapters Layer (어댑터 계층)
- **책임**: 포트 인터페이스를 구현하여 외부 시스템과 통신
- **특징**: 구체적인 기술 구현
- **주요 어댑터**:
  - Inbound: CLI (Typer), Web API (FastAPI)
  - Outbound: Dataset loaders (CSV/Excel/JSON), LLM adapters (OpenAI/Anthropic/Ollama/vLLM), Storage adapters (SQLite/PostgreSQL), Tracker adapters (Langfuse/MLflow/Phoenix)

## 3. 전문가 관점 기반 정의

### 3.1 전문가 그룹별 핵심 관점

| 전문가 그룹 | 핵심 관심사 | EvalVault 적용 포인트 |
|------------|------------|---------------------|
| **인지심리학자** | 인지 부하 최소화, 패턴 인식, 작업 기억 보조 | 점진적 정보 공개, 시각적 그룹핑, 컨텍스트 전환 최소화 |
| **문서 편집자** | 명확한 의사소통, 정보 구조, 가독성 | README, ARCHITECTURE, 가이드 문서 체계화 |
| **뇌과학자** | 시각 처리 최적화, 인지 편향 활용 | 시각적 대비, 패턴 인식 촉진, anchoring 프레이밍 |
| **정보공학 전문가** | 정보 아키텍처, 데이터 구조화, 확장성 | Run → Test Case → Stage → Metric 계층 |
| **교육공학 전문가** | 학습 효과, 점진적 학습, 피드백 루프 | 초급/중급/고급별 정보 제공, 툴팁, 예시 |
| **UI/UX 전문가** | 워크플로우 최적화, 피드백, 접근성 | 평가 실행 → 결과 확인 → Phoenix 분석 흐름 |
| **정보컴퓨팅 전문가** | 알고리즘 효율성, 데이터 처리 최적화 | AsyncBatchExecutor, StreamingDatasetLoader |
| **소프트웨어 개발 전문가** | 성능, 확장성, 유지보수성, 테스트 가능성 | 의존성 주입, SOLID 원칙, 포트 기반 설계 |
| **아키텍트** | 시스템 아키텍처, 확장성, 표준 준수 | Hexagonal Architecture, OpenTelemetry 표준 |

### 3.2 통합 적용 원칙

```
┌─────────────────────────────────────────────────────────────────────────────────────┐
│              전문가 관점 통합 적용 원칙 (EvalVault 설계 철학)                      │
├─────────────────────────────────────────────────────────────────────────────────────┤
│                                                                              │
│  1. 인지 부하 최소화 (인지심리학) + 정보 아키텍처 (정보공학)                     │
│     → 계층적 정보 구조: Run → Test Case → Stage → Metric                         │
│                                                                              │
│  2. 패턴 인식 촉진 (인지심리학) + 시각적 대비 (뇌과학)                         │
│     → 색상 인코딩: 성공(녹색), 경고(노란색), 실패(빨간색), 정보(파란색)           │
│                                                                              │
│  3. 워크플로우 최적화 (UI/UX) + 점진적 정보 제공 (교육공학)                      │
│     → 단계적 노출: 요약 → 상세 → Phoenix 고급 분석                                │
│                                                                              │
│  4. 성능 최적화 (정보컴퓨팅) + 확장성 (아키텍트)                               │
│     → AsyncBatchExecutor, StreamingDatasetLoader, 포트 기반 확장                 │
│                                                                              │
│  5. 명확한 의사소통 (문서 편집자) + SOLID 원칙 (소프트웨어 개발)                │
│     → 포트 인터페이스 계약, 명확한 책임 분리, 체계화된 문서화                   │
│                                                                              │
└─────────────────────────────────────────────────────────────────────────────────────┘
```

## 5. 작성 방법론

### 5.1 문서화 원칙

| 원칙 | 설명 | 적용 예시 |
|------|------|----------|
| **명확성** | 모호한 표현 피하기 | "빠른 속도" → "P95 < 1초" |
| **일관성** | 용어, 형식, 구조 통일 | 모든 메트릭: 정의, 중요성, 계산 방식, 임계값 |
| **계층성** | 개요 → 상세 → 심화 순서 | 제1부(개요) → 제4부(컴포넌트 상세) → 제8부(고급 기능) |
| **실용성** | 실제 사용 가능한 예시 | CLI 명령 예시, 코드 예시, 설정 파일 예시 |
| **확장성** | 향후 확장 가능한 구조 | 포트 인터페이스 기반 확장 지점 명시 |

각 섹션에 해당 전문가 관점을 명시하고, 실제 설계/구현에 어떻게 적용되었는지 설명:

```

### 5.1.1 인지 부하 최소화

**인지심리학적 근거**:
- 인간의 작업 기억 용량은 제한적 (7±2 청크)
- 과도한 정보는 인지 부하 증가 → 오류율 증가

**EvalVault 적용**:
1. 점진적 정보 공개 (Progressive Disclosure)
   - 기본: 요약 메트릭 + 핵심 인사이트
   - 확장: 클릭/호버로 상세 정보 표시
   - 심화: Phoenix 링크로 전문 분석

2. 시각적 그룹핑
   - 성공/실패 케이스를 색상으로 즉시 구분
   - 클러스터별로 공간적으로 그룹화
   - 단계별로 시각적 구분 (Retrieval → Rerank → Generation)

**실제 구현**:
- `RunDetails.tsx`에서 요약 → 상세 → Phoenix 링크 순서로 노출
- 차트 색상: 성공(Green 500-600), 경고(Yellow 400-500), 실패(Red 500-600)
- Phase별 시각화: Input(Gray), Retrieval(Blue), Output(Green)

```

## 8. 리스크 및 완화 전략

### 8.1 주요 리스크

| **기술 변경 추적 실패** | 중 | 중 | 최신 코드 기반 작성, 버전 관리 |
| **일관성 유지 어려움** | 높 | 중 | 자동화 도구 사용, 스타일 가이드 준수 |
| **기한 초과** | 낮 | 높 | 우선순위 기반 작업, MVP 먼저 완료 |

### 8.2 완화 전략 상세

1. **프로젝트 규모 관리**
   - MVP(최소 실행 가능 제품) 우선 완료
   - 우선순위 기반(P0 > P1 > P2) 작업
   - 반복적 개선(Iterative improvement)

2. **전문가 관점 통합**
   - 초기 합의 문서 작성 (본 계획 문서)
   - 정기 리뷰 미팅 (주간)
   - 충돌 시 우선순위 기반 의사결정

3. **기술 변경 추적**
   - 최신 코드 기반 작성 (main 브랜치)
   - 버전 관리 (Git 태그)
   - 변경 사항 기록 (CHANGELOG.md 참조)

4. **일관성 유지**
   - 스타일 가이드 준수 (`docs/internal/reference/STYLE_GUIDE.md`)
   - 자동화 도구 사용 (Ruff, Pytest, Lint)
   - 정리 리뷰(Review) 필수

5. **기한 관리**
   - 데일리 스탠드업 (진행 상황 공유)
   - 주간 마일스톤 검수
   - 리스크 조기 감지 및 대응

## 결론

본 계획 문서는 EvalVault 프로젝트의 전체 분석 결과를 바탕으로, 다양한 전문가 관점을 통합한 종합 개발 백서 작성을 위한 청사진을 제공합니다.

**핵심 목표**:
1. EvalVault의 목적과 구현 방법을 명확히 설명
2. 아키텍처, 클래스, 연결 고리, 구현 방법의 효과를 상세히 기술
3. 사용 방법과 확장 방법을 제시
4. 전문가 관점을 통합하여 실용적이고 이해하기 쉬운 문서 작성

**다음 단계**:
1. 세부 문서 작성 시작
2. 정기 리뷰 및 피드백 수집
3. 최종 통합 및 품질 검수
4. 배포 및 유지보수

이 계획이 성공적으로 실행되면, EvalVault는 RAG 평가 플랫폼의 표준 가이드로 자리 잡고, 개발자와 사용자에게 명확하고 유용한 문서를 제공할 수 있을 것입니다.
